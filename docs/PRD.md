# 📄 Product Requirements Document (PRD)

## 🧠 Product Name  
**SupplyMind – Autonomous Supply Chain Coordinator**

---

# 1. 🎯 Product Vision

SupplyMind is an **Agentic AI-powered, microservices-based platform** that autonomously manages supply chain and procurement operations for large FMCG enterprises.

It combines:
- Multi-agent orchestration (LangChain + LangGraph)
- Retrieval-Augmented Generation (RAG)
- Vector-based knowledge systems
- Human-in-the-loop governance
- Production-grade observability and guardrails

The system is designed to be:
- **Kubernetes-native**
- **Composable as microservices**
- **Runnable locally via Docker Compose**

---

# 2. 🎯 Objectives

## Primary Goals
- Reduce manual procurement effort by **60–80%**
- Enable **AI-assisted decision-making with reasoning transparency**
- Prevent stockouts and overstocking
- Optimize vendor selection dynamically

## Engineering Goals
- Cloud-native (Kubernetes-ready)
- Observable (metrics, traces, logs)
- Safe (guardrails + human approval)
- Extensible (agent + tool architecture)

---

SupplyMind is built as a **distributed microservices system**, where each component is independently deployable.

# 4. 🤖 Core AI Stack

---

## 4.1 Retrieval-Augmented Generation (RAG)

SupplyMind uses RAG to enhance decision-making with enterprise knowledge.

### Sources
- Vendor contracts
- Historical POs
- Pricing agreements
- SLA documents
- Email communications

### Flow
1. Query generated by agent
2. Embedding created
3. Retrieved from Vector DB
4. Injected into LLM context
5. Used for reasoning

---

## 4.2 Vector Database

### Purpose
- Store embeddings for:
  - Vendor knowledge
  - Historical decisions
  - Procurement patterns

### Suggested Options
- Pinecone / Weaviate / Chroma / FAISS

### Example Use Cases
- Retrieve best past vendor for similar scenario
- Compare historical pricing trends

---

## 4.3 ELK Stack (Centralized Logging & Analysis)

SupplyMind uses the **ELK Stack (Elasticsearch, Logstash, Kibana)** for **centralized logging, monitoring, and analysis across all microservices and AI agents**.

This is critical in an Agentic AI system where:
- Multiple services interact asynchronously  
- LLM decisions need traceability  
- Debugging requires full visibility across workflows  

---

### Components

#### Elasticsearch
- Stores all logs in a scalable, searchable format  
- Supports fast querying across:
  - Agent decisions  
  - API calls  
  - Errors and exceptions  
  - Workflow execution traces  

---

#### Logstash
- Collects and processes logs from:
  - Agent microservices  
  - Orchestrator (LangGraph)  
  - API Gateway  
  - External integrations  

- Performs:
  - Log parsing  
  - Enrichment (add metadata like agent name, workflow ID)  
  - Transformation into structured format  

---

#### Kibana
- Visualization and analysis layer  
- Provides dashboards for:
  - System health  
  - Agent activity  
  - Error trends  
  - Workflow timelines  

---

### What is Logged

#### System Logs
- Service startup/shutdown  
- API requests/responses  
- Inter-service communication  

#### AI/Agent Logs
- Agent inputs and outputs  
- Decision reasoning summaries  
- RAG retrieval context  
- Confidence scores  

#### Workflow Logs
- End-to-end execution traces  
- Step-by-step agent transitions  
- State changes in LangGraph  

#### Error Logs
- Failed API calls  
- Vendor communication issues  
- Guardrail violations  

---

### Key Use Cases

#### 1. Debugging AI Decisions
- Trace why a specific vendor was selected  
- Inspect retrieved RAG context  
- Identify incorrect reasoning  

---

#### 2. Observability Across Microservices
- Track request flow across services  
- Identify bottlenecks and latency issues  

---

#### 3. Audit & Compliance
- Maintain logs of:
  - PO decisions  
  - Human approvals  
  - System actions  

---

#### 4. Incident Analysis
- Detect patterns in failures  
- Analyze recurring vendor issues  
- Monitor system anomalies  

---

### Integration with Other Observability Tools

- **Langfuse** → LLM-specific tracing (prompts, responses, token usage)  
- **Prometheus + Grafana** → Metrics (latency, CPU, throughput)  
- **ELK** → Logs and deep analysis  

👉 Together, they provide **full-stack observability**:
- Metrics (what is happening)  
- Logs (why it happened)  
- Traces (how it happened)  

---

### Deployment

- Runs as separate services in:
  - Docker Compose (local development)  
  - Kubernetes (production via StatefulSets)  

- Integrated via:
  - Fluent Bit / Filebeat (log shippers)

# 5. 🤖 Agent Architecture (Enhanced)

Each agent runs as a **separate microservice**.

---

## 5.1 Inventory Monitoring Agent
- Publishes stock events to message queue
- Writes to shared state store

---

## 5.2 Demand Forecasting Agent
- Uses ML models
- Stores forecasts in DB + vector embeddings

---

## 5.3 Vendor Intelligence Agent
- Uses RAG + EKL
- Retrieves:
  - Past vendor performance
  - Contract terms
- Outputs **ranked vendors + reasoning**

---

## 5.4 Procurement Agent
- Core decision engine
- Combines:
  - Forecast
  - Inventory
  - Vendor insights (via RAG)

### Outputs
- PO draft
- Detailed reasoning trace

---

## 5.5 Logistics / Tracking Agent
- Polls ERP/vendor APIs
- Uses event-driven triggers

---

## 5.6 Exception Handling Agent
- Subscribes to anomaly events
- Re-triggers procurement decisions

---

## 5.7 Human Approval Agent
- UI + API layer
- Stores approval logs
- Sends feedback to learning loop

---

# 6. 🔁 Key Workflows (Enhanced with RAG + Guardrails)

---

## 6.1 Low Inventory → PO → Approval

1. Inventory event triggered  
2. Forecast agent predicts shortage  
3. Procurement agent:
   - Queries EKL via RAG  
   - Retrieves similar past decisions  
4. Vendor agent ranks vendors  
5. Guardrails validate decision  
6. PO generated with reasoning  
7. Human approval required  
8. Execution triggered  

---

## 6.2 Vendor Selection (RAG-driven)

Decision uses:
- Historical vendor success
- Contract terms (via RAG)
- Pricing trends

### Example Reasoning

Selected Vendor B because:

- Delivered 98% on-time in last 20 orders
- Contract allows priority dispatch
- Price within 5% of lowest vendor
- Past similar scenario showed best outcome


---

## 6.3 Exception Handling

- Delay detected  
- RAG retrieves:
  - Past delay resolutions  
- Suggests:
  - Alternate vendor  
  - Expedite option
